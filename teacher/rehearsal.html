개강 전 모의 리허설 가이드

B·E 두 학급의 실제 코스를 전혀 건드리지 않고, 별도의 “모의 코스” 하나에서 과제 생성 → 제출물 수집 → 채점 초안까지 파이프라인 전체를 미리 돌려볼 수 있습니다. 대상 파일은 Mock.gs이며, 교사가 직접 실행하는 함수는 createMockCourse() · runMockDrill() · teardownMockCourse() 세 가지입니다.

🔁 5단계 한눈 요약
  • ① 모의 코스 생성 → ② (선택) 학생 역할 계정 초대 → ③ 리허설 드릴 실행 → ④ 확인 지점 점검 → ⑤ 정리
💡 DRY_RUN 리허설과 무엇이 다른가

testConnection()과 DRY_RUN=true 상태의 createAssignments() 등은 “무엇을 할 예정인지”만 로그로 보여줄 뿐, 실제 Classroom API 호출(코스 생성·활성화·과제 생성·제출물 조회·채점 patch)이 우리 학교 Google Workspace 환경에서 정말 성공하는지는 확인해주지 않습니다. 이 모의 리허설은 그 실행 대상을 실코스가 아닌 “모의 코스” 하나로 국한해, API 호출을 실제로 실행하며 안전하게 검증합니다. Code.gs의 DRY_RUN 값과는 무관하게 항상 동작합니다 — 대상이 애초에 실코스가 아니기 때문입니다.

🎬 5단계 절차

  1. 1
    createMockCourse()

    [모의] 데이터과학 리허설이라는 이름의 코스를 새로 만들고 즉시 ACTIVE로 전환합니다. 이미 만든 모의 코스가 있으면 재사용해 중복 생성을 막습니다.

    실행 후 로그에 뜨는 클래스룸 링크를 열어 코스가 실제로 생겼는지 확인합니다.

  2. 2
    클래스룸 화면에서 학생 역할 계정 초대 선택

    동료 교사 계정이나 테스트 계정을 학생으로 초대합니다. 초대해야 수집·채점 단계까지 실데이터로 검증됩니다 — 초대하지 않아도 다음 단계는 실행되지만 검증 범위가 제한됩니다. 자세한 조건은 아래 “학생 역할 시뮬레이션 안내”를 참고합니다.

  3. 3
    runMockDrill()

    모의 과제 2개(URL형 1개 + 서술형 1개)를 만들고, 제출물 수집(studentSubmissions.list)과 — 제출물이 있다면 — draftGrade 채점까지 한 번에 실행합니다.

    여러 번 실행해도 안전합니다. 과제 설명文 끝의 숨은 표시자(marker)로 중복 생성을 막기 때문입니다.

  4. 4
    확인 지점

    다음 세 곳을 직접 눈으로 확인합니다.

    • 클래스룸 모의 코스 화면에 [모의] 접두 과제 2개가 보이는지
    • 관리 스프레드시트의 모의_리허설_로그 시트에 단계별 기록이 쌓였는지
    • 학생을 초대했다면 클래스룸에서 draftGrade(99점)가 반영되어 보이는지
  5. 5
    teardownMockCourse()

    모의 코스를 ARCHIVED로 전환하고 저장된 모의 코스 속성을 지워 정리합니다.

    완전히 삭제하고 싶다면(되돌릴 수 없음) deleteMockCourse()를 별도로 실행합니다 — ARCHIVED 상태가 아니면 실패합니다.

🎭 학생 역할 시뮬레이션 안내

🙋 누구를 학생으로 초대할까
  • 개인 Gmail 계정은 초대할 수 없습니다. 학교 Education 도메인 코스에는 개인 Gmail 계정이 도메인 정책과 무관하게 초대 대상에서 배제됩니다.
  • 권장: 같은 도메인의 별도 계정을 학생으로 초대합니다 — 예: 동료 교사 계정, 또는 관리자가 발급한 테스트 계정. 클래스룸 화면에서 수동으로 초대합니다(Mock.gs는 초대를 자동화하지 않습니다).
  • 교사 본인 계정은 자신이 만든 코스에 학생으로 겸직할 수 없습니다(ALREADY_EXISTS 오류). 반드시 별도 계정을 씁니다.

🔍 검증 범위 — 정직하게

⚠️ 학생 0명이면 채점·반환은 미검증으로 남습니다

학생 없이도 되는 것: 코스 생성·PROVISIONED→ACTIVE 전환, 과제 생성(중복 방지 포함), studentSubmissions.list 조회(학생 0명이면 빈 배열이 정상 — 오류가 아닙니다), 코스 정리(ARCHIVED).

학생을 초대해야 되는 것: 제출물이 실제로 채워진 studentSubmissions.list 결과, draftGrade patch 검증(제출물이 있어야 patch 대상이 생깁니다).

runMockDrill()이 자동 실행하지 않는 것: 학생 반환(return). draftGrade patch까지만 자동 검증하고, 반환은 클래스룸 화면에서 직접 확인하거나 별도로 실행해야 합니다 — 되돌릴 수 없는 동작이므로 리허설이라도 자동화하지 않습니다.

🛡️ 안전장치 — 실코스 오염 방지

✅ 실코스는 절대 건드리지 않습니다
  • Mock.gs는 실코스 저장 속성을 절대 읽거나 쓰지 않습니다 — 별도의 PROP_KEY_MOCK_COURSE_ID만 사용합니다.
  • 모든 진입점 함수(createMockCourse() · runMockDrill() · teardownMockCourse() · deleteMockCourse())는 Classroom에 쓰기 요청을 보내기 전에 assertNotRealCourse_()로 대상이 실코스 저장 속성과 겹치는지 확인하고, 겹치면 즉시 중단합니다.
  • Code.gs · Collect.gs · Grades.gs의 기존 함수는 전혀 수정하지 않습니다. Mock.gs는 그 함수들을 호출만 합니다.

❓ 자주 묻는 것

Q 모의 코스를 완전히 지우고 싶어요.

teardownMockCourse()는 ARCHIVED 전환과 속성 삭제까지만 합니다. 완전 삭제(되돌릴 수 없음)를 원하면 deleteMockCourse()를 별도로 실행해야 하며, 코스가 ARCHIVED 상태가 아니면 실패합니다 — 반드시 teardownMockCourse()를 먼저 실행한 뒤 시도합니다.

Q runMockDrill()을 여러 번 실행해도 괜찮나요?

네. 과제 설명文 끝에 숨겨 둔 표시자(marker)로 중복 생성을 막기 때문에, 여러 번 실행해도 모의 과제가 두 번 생기지 않습니다.

Q DRY_RUN=true인 상태에서도 모의 리허설을 실행할 수 있나요?

네. Mock.gs의 함수들은 Code.gs의 DRY_RUN 값과 무관하게 항상 실제로 씁니다. 대상이 실코스가 아닌 별도의 모의 코스로 한정되어 있고 assertNotRealCourse_()가 이를 재확인하므로, 실코스 쓰기를 막아 둔 DRY_RUN=true 상태에서도 모의 코스 리허설은 지장 없이 진행할 수 있습니다.
